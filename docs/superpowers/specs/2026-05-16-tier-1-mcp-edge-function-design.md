# Tier 1 — MCP as Supabase Edge Function (HTTP transport)

**Date:** 2026-05-16
**Type:** Backend architecture (Tier 1; ground floor for Tier 2)
**Status:** Drafted; pending user review

## Problem

The Plannen MCP server at `mcp/src/index.ts` is a Node process that speaks the **stdio** MCP transport, spawned as a subprocess by Claude Code via the plugin manifest at `plugin/.claude-plugin/plugin.json`. This shape ties the MCP to the user's local machine and to a single OS process. The shape works for Tier 0 and the default Tier 1 experience, but it blocks two things we want next:

1. **Tier 2 (Self-Hosted Cloud)** needs the MCP server to live in the user's Supabase Cloud project as a deployed Edge Function, reachable over HTTPS from any MCP-aware agent on the network. Stdio cannot leave the host.
2. **Tier 1 dev ergonomics**: building Tier 2 confidently requires a local test bed for the cloud-shaped MCP, served by `supabase functions serve` against the existing local Supabase Docker stack. Without that, Tier 2's MCP code is only exercised in production.

This spec defines Phase A: the **port of the MCP server to a Supabase Edge Function** under `supabase/functions/mcp/`, running locally in Tier 1 via `supabase functions serve mcp`. Phase B (Tier 2 cloud deployment, separate spec) `supabase functions deploy mcp`s the same function to Supabase Cloud as one step of a larger cloud-bootstrap flow.

## Decision

Add a new MCP server implementation at `supabase/functions/mcp/`, written in Deno, speaking MCP's `StreamableHTTPServerTransport`, sharing the same `_shared/db.ts` Postgres pool and `withDb` user-context pattern as the existing edge functions. The Node MCP at `mcp/src/` stays unchanged and remains the default for both Tier 0 and Tier 1. Users opt into the HTTP MCP via `scripts/mcp-mode.sh http`, which rewrites the plugin manifest in place.

**Tier scoping:** Tier 1 only in Phase A. Tier 0 has no Supabase services and continues to use the Node stdio MCP exclusively. Tier 2 (cloud) uses the same Deno function deployed remotely — that wiring is Phase B's concern.

**Auth model in Phase A:** single user via shared `MCP_BEARER_TOKEN`. The bearer is generated by `scripts/mcp-mode.sh http` on first switch, persisted in `supabase/.env.local` (read by `supabase functions serve`) and in the rewritten `plugin.json` `headers.Authorization`. **Multi-user is explicitly out of scope for Phase A** and is captured in [Future work](#future-work).

## Architecture

Phase A adds a parallel MCP implementation. Existing flows (Tier 0, Tier 1-stdio) are unaffected.

```
┌─── Tier 0 ──────────────────────┐  ┌─── Tier 1 stdio (default) ──────┐
│ Claude Code → plannen plugin    │  │ Claude Code → plannen plugin    │
│   ↓ stdio                       │  │   ↓ stdio                       │
│ node mcp/dist/index.js          │  │ node mcp/dist/index.js          │
│   ↓ pg                          │  │   ↓ pg                          │
│ embedded pg @ 54322             │  │ Supabase Docker pg @ 54322      │
└─────────────────────────────────┘  └─────────────────────────────────┘

┌─── Tier 1 HTTP (Phase A — opt-in) ────────────────────────┐
│ Claude Code → plannen plugin                              │
│   ↓ HTTPS (loopback) + Bearer                             │
│ http://127.0.0.1:54321/functions/v1/mcp                   │
│   = supabase/functions/mcp/ via `supabase functions serve`│
│   ↓ _shared/db.ts (npm:pg@8 + withDb)                     │
│ Supabase Docker pg                                        │
└───────────────────────────────────────────────────────────┘
```

The HTTP MCP and stdio MCP are never both active for the plugin simultaneously — the plugin manifest carries one MCP server entry, and `mcp-mode.sh` swaps which one it points to.

## Components

| File / area | Change |
|---|---|
| `supabase/functions/mcp/index.ts` *(new)* | Edge Function entry. Validates `Authorization: Bearer <MCP_BEARER_TOKEN>`. Instantiates `StreamableHTTPServerTransport` from `@modelcontextprotocol/sdk`, hands it the server from `server.ts`. |
| `supabase/functions/mcp/server.ts` *(new)* | Constructs the MCP `Server` instance, registers tool handlers from `tools/*.ts`. Resolves the bootstrap user once at module load via the existing `PLANNEN_USER_EMAIL` env (single-user, mirrors `mcp/src/index.ts`). |
| `supabase/functions/mcp/tools/*.ts` *(new, multiple files)* | Tool definitions split by domain to avoid a single 2,000-line file: `events.ts`, `memories.ts`, `stories.ts`, `profile.ts`, `relationships.ts`, `locations.ts`, `sources.ts`, `watches.ts`, `rsvp.ts`, `groups.ts`, `wishlist.ts`, `family.ts`. Each exports a `register(server, ctx)` function. Mirrors the logical tool groups already present in `mcp/src/index.ts`. |
| `supabase/functions/mcp/deno.json` *(new)* | Deno config + npm-specifier import map for `@modelcontextprotocol/sdk`, `pg@8`, `zod@3`. Aligns versions with the rest of `supabase/functions/`. |
| `supabase/functions/_shared/db.ts` | Unchanged. The MCP function imports `withDb` directly. |
| `supabase/functions/mcp/tools/*.test.ts` *(new)* | Deno tests for each tool module. Pure-logic tests ported from `mcp/src/*.test.ts` where they translate cleanly (recurrence, profileFacts, sources). |
| `tests/smoke/tier1-http-mcp.sh` *(new)* | Bash smoke test: `supabase start` → `mcp-mode.sh http` → `supabase functions serve mcp` (bg) → `curl` JSON-RPC `tools/list`, expect >40 tools → teardown. |
| `scripts/mcp-mode.sh` *(new)* | `stdio` writes the existing Node-stdio plugin.json entry. `http` generates a 32-byte bearer (idempotent: reuses existing if `MCP_BEARER_TOKEN` already present in `supabase/.env.local`), writes the bearer to `supabase/.env.local`, rewrites `plugin/.claude-plugin/plugin.json` `mcpServers.plannen` to `{ "type": "http", "url": "http://127.0.0.1:54321/functions/v1/mcp", "headers": { "Authorization": "Bearer <token>" } }`. Prints "reload the plannen plugin in Claude Code to apply." |
| `plugin/.claude-plugin/plugin.json` | Default content unchanged (stdio). Edited in place by `mcp-mode.sh` when switching modes. The git-checked-in version stays stdio. |
| `plugin/commands/plannen-doctor.md` | Add a "MCP mode: stdio / http" detection line based on the current plugin.json shape. |
| `mcp/src/*.ts` (Node) | **Unchanged.** Tier 0 and Tier 1-stdio keep using this verbatim. The Node tests stay alive. |

**Tool count and parity.** `mcp/src/index.ts` registers ~50 tools today. Phase A ports all of them **except `transcribe_memory`** — the Node version spawns local `whisper.cpp` via `child_process.spawn`, which Deno Edge Functions cannot do. The HTTP MCP's `tools/list` response will not contain `transcribe_memory`. This asymmetry is documented in `README.md` and `plannen-doctor` output. Phase A does not add a cloud-API replacement; that is a separate decision when there's real demand.

**File split during the port.** The Node MCP keeps all tool definitions in one 2,121-line `index.ts`. The Deno port splits them into `tools/<domain>.ts` files at port time — same logical grouping, same SQL, same response shapes, but reviewable in chunks. This is *during* the port, not a separate refactor pass.

## Data flow

**Dev flow — switching to HTTP and running:**

```
$ bash scripts/local-start.sh                             # local Supabase up
$ bash scripts/mcp-mode.sh http                           # generates bearer, rewrites plugin.json
$ supabase functions serve mcp                            # serves the function locally
# (in Claude Code, reload the plannen plugin)
> User: "list my events"
            ↓
   Plugin MCP client (HTTP) issues JSON-RPC tools/call
            ↓
   POST http://127.0.0.1:54321/functions/v1/mcp
   Authorization: Bearer <MCP_BEARER_TOKEN>
            ↓
   Edge Function validates bearer (constant-time compare)
            ↓
   StreamableHTTPServerTransport routes JSON-RPC to MCP Server
            ↓
   Tool handler in tools/events.ts
            ↓
   withDb(USER_ID, async (client) => { ... SQL ... })
            ↓
   pg.Pool checkout → BEGIN → set request.jwt.claim.sub + app.current_user_id GUCs → query → COMMIT → release
            ↓
   JSON-RPC response → HTTP body → plugin → Claude
```

**Switching back to stdio:**

```
$ bash scripts/mcp-mode.sh stdio
# (in Claude Code, reload the plannen plugin)
```

`supabase/.env.local` retains the bearer. Next `mcp-mode.sh http` reuses it (the token doesn't rotate on every toggle).

## Schema changes

**None in Phase A.** The HTTP MCP runs against the existing `plannen.*` schema with the existing RLS policies. No new tables, no migrations.

Phase B (Tier 2 cloud deploy) also adds no schema changes; the functions just run elsewhere.

Phase A.1 (multi-user — see [Future work](#future-work)) introduces `plannen.api_tokens` — that migration lives in *its* spec, not this one.

## Error handling

| Failure mode | Response |
|---|---|
| Missing `Authorization` header | 401 JSON `{ "error": "missing_bearer" }` |
| Bearer doesn't match `MCP_BEARER_TOKEN` | 401 JSON `{ "error": "invalid_bearer" }`. Constant-time compare; same response for unknown vs malformed token. |
| `DATABASE_URL` unreachable / pool exhausted | JSON-RPC error frame, code `-32000`, `message: "database_unavailable"`. Logged to function stderr. |
| Tool handler throws | JSON-RPC error frame with the thrown error's `.message`. Stack trace logged but not returned. Mirrors Node MCP behaviour. |
| Request exceeds Edge Function time limit (~50s for `serve`, 150s for cloud) | Handler must `client.release()` before timeout. Long tools (`list_event_memories` with many photos) already paginate; no change. |
| `mcp-mode.sh http` re-run | Idempotent: reuses existing `MCP_BEARER_TOKEN` from `supabase/.env.local` if present; otherwise generates new. `plugin.json` rewrite is overwrite-safe. |
| `mcp-mode.sh stdio` while `supabase functions serve` is still running | No-op on the function process. Only `plugin.json` is rewritten. User restarts the plugin in Claude Code. |
| `transcribe_memory` invoked over HTTP MCP | Not registered → MCP returns "tool not found" via the SDK's standard error. Documented in `plannen-doctor`. |

## Testing

Three layers, smallest to largest:

| Layer | What | Where |
|---|---|---|
| **Unit (Deno)** | `tools/<domain>.test.ts` for each tool module. Pure-logic tests ported from `mcp/src/*.test.ts` where they translate cleanly (recurrence math, profileFact confidence scoring, source URL parsing). Shape-only checks for tools that hit the DB (verify response field names match the Node version). | `deno test supabase/functions/mcp/` |
| **Integration** | Spin up local Supabase + run `supabase functions serve mcp` + hit it via `fetch` with JSON-RPC bodies. Cover: `tools/list` count, one representative tool from each domain group, bearer-auth rejection (401 on missing/wrong). | `vitest`, gated by `RUN_INTEGRATION=1` (skipped in default CI). |
| **Smoke** | `tests/smoke/tier1-http-mcp.sh` — full dev flow end-to-end: `local-start.sh` → `mcp-mode.sh http` → `supabase functions serve mcp` (background) → `curl` `tools/list` with bearer → expect >40 tools → tear down. Run before each Phase A PR. | Bash, manual + can be wired to CI later. |

**Existing Node MCP tests stay alive** for `mcp/src/*` — they cover Tier 0 and Tier 1-stdio. The Deno port does not re-prove the SQL semantics tool-by-tool; the schema and queries are unchanged, and the Node tests already cover them. The Deno tests focus on the port's shape (correct tool registration, correct response envelopes, transport behaviour).

**CI integration.** Add a `deno test` job to the existing GitHub Actions workflow alongside the Node test jobs. Integration + smoke stay opt-in (gated env var) — they require Docker + the full Supabase stack, which is too heavy for default CI.

## Out of scope (Phase A)

- **Multi-user auth.** Single bearer for the bootstrap user. See [Future work](#future-work).
- **Tier 0 HTTP MCP.** Tier 0 has no Supabase Edge Functions; an HTTP MCP there would need a dedicated Node route in `backend/`. Not blocked by Phase A, but not addressed by it either.
- **Tier 2 cloud deploy.** That orchestration (`supabase link` / `db push` / `functions deploy` / `secrets set` / photo upload / plugin reconfig to a cloud URL) is Phase B and lives in its own spec.
- **Cloud-API transcription** (replacement for `transcribe_memory` in the HTTP MCP). Deferred until there's a real user request.
- **Web UI for MCP config** (token mint/rotate, mode switch). CLI helper is sufficient for Phase A's audience.
- **OAuth for MCP.** Bearer is sufficient for single-user. Multi-user phase revisits.

## Future work

- **Phase A.1 — Multi-user HTTP MCP.** New `plannen.api_tokens` table (token text PK, user_id, name, created_at, last_used_at) with RLS so users only see their own. MCP middleware swaps from boot-time env user to per-request token-derived user. CLI helper `scripts/mcp-token-create.sh --email <user>` mints tokens for Tier 1 dev. Web UI for token management follows when the cloud web app ships.
- **Phase B — Tier 2 cloud deploy.** Bootstrap + migration scripts that `supabase functions deploy mcp` + the rest of the stack to a user's Supabase Cloud project. Uses the function built in this Phase A spec verbatim.
- **Tier 0 HTTP MCP.** Dedicated Node HTTP route in `backend/src/routes/mcp.ts` exposing the same tool surface for users on Tier 0 who want HTTP without Supabase. Only needed if real demand emerges.
- **Cloud transcription.** Replace dropped `transcribe_memory` with an OpenAI Whisper / AssemblyAI / Anthropic-future-audio call. Adds an external dependency and config; only build when requested.

## Open questions / risks

- **`@modelcontextprotocol/sdk` Deno compatibility.** The SDK ships ESM and supports Deno via `npm:` specifiers in practice, but the `StreamableHTTPServerTransport` implementation has only been widely exercised on Node. Risk: undocumented Node-only assumption surfaces during port. Mitigation: prototype the transport hookup on day 1 of the plan; if blocked, fall back to a thin handwritten transport (the JSON-RPC framing is ~100 LOC).
- **`pg` driver behaviour under Supabase Edge Function's `Deno.serve`.** The `_shared/db.ts` `Pool` works for the existing 11 edge functions, but the MCP function is more chatty (many concurrent tool calls in a single conversation). Risk: pool exhaustion under load. Mitigation: cap `Pool` size in `_shared/db.ts` config and surface clean errors when exhausted; revisit if the MCP function actually hits the cap.
- **Bearer in `plugin.json` checked into git.** `mcp-mode.sh http` writes the bearer to a file that is *not* checked in by default (the plugin manifest is per-install). Risk: a user accidentally commits a populated `plugin.json`. Mitigation: `.gitignore` entry covering the populated form, plus a `mcp-mode.sh stdio` step in any "prepare for commit" docs.
- **Plugin reload friction.** Switching modes requires the user to reload the plannen plugin in Claude Code. There's no programmatic reload. Risk: confusion about why a switch "didn't take." Mitigation: `mcp-mode.sh` prints the reload instruction prominently; `plannen-doctor` flags mismatch between configured and active mode.
